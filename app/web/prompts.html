The Hook
45 seconds
Screen: Splash page or Interactive Demo hero
EDT builds personalized day-by-day travel plans. What makes it different: no LLM. No GPT, no Claude, no Gemini. It runs a RAG pipeline backed by pgvector cosine similarity search to match travelers with the right places based on their persona.
The entire system is retrieval + scoring + assembly. We retrieve POIs using vector search, score them against the traveler's persona, and assemble day-by-day itineraries algorithmically. All running on FastAPI + async PostgreSQL, deployed on Google Cloud Run.
Point out: Tech badges on the demo page — pgvector, 384-dim embeddings, FastAPI, Cloud Run

Anonymous Session & Smart Resume
60 seconds
Screen: Interactive Demo → Step 1 (Register Session)
Click "Register Session" button on demo page
No signup needed. The app generates a random UUID on first launch, stores it locally, and sends it as an X-Device-ID header on every request. The server creates a DeviceSession record. Everything persists across app restarts without forcing login.
POST /api/v1/sessions/register
Header: X-Device-ID: demo-abc123...
Response: { "device_id": "...", "platform": "web-demo" }
Click "Seed Demo Data" (Step 2) — creates a trip draft + 4 wishlist landmarks
The demo seed pre-populates a trip draft — Italy, 10 days, couple, with cultural, romantic, and foodie vibes selected. It also adds 4 iconic landmarks to the wishlist: Colosseum, Eiffel Tower, Fushimi Inari Shrine, and Sagrada Familia. All with images.
POST /api/v1/demo/seed
Response: { "status": "seeded", "draft_id": "...", "wishlist_items_added": 4 }
Click "Get Active Draft" (Step 3)
This is the auto-resume system. Every time the app launches, it checks for an active draft. If one exists, it shows a "Resume your trip" banner. The draft stores the wizard step, country, dates, vibes, budget — everything needed to pick up exactly where you left off.
Point out in the JSON: current_step: 2, country_name: "Italy", vibes: ["cultural", "romantic", "foodie"], status: "active"

Persona Understanding
90 seconds
Screen: Interactive Demo → Step 5 (Countries) + Step 6 (Allocations)
Click "Load Countries" (Step 5)
The country config drives the entire planning pipeline. We have 5 countries with 18 cities. Each city has min/max days, highlights, vibes, and a priority score. The country planner uses this metadata to allocate days across cities based on the traveler's persona.
Point out: Each city has vibes, min_days, max_days, highlights — this is the constraint graph
Click "Plan Italy Trip" (Step 6)
The country planner takes the persona — couple, cultural, romantic, foodie — and runs constraint satisfaction to allocate days across Italian cities. It generates 2 options with match scores. The score is computed from how well each city's POIs align with our vibes, using persona vectors stored in the pgvector database.
No LLM is involved here. This is pure algorithmic planning: city metadata + persona multipliers + constraint satisfaction. The 87% match means 87% of the city's top POIs align with our cultural-romantic-foodie persona.
Point out: Two allocation options with different strategies. City names, day counts, match scores. "Recommended" badge on the higher-scoring option.
POST /api/v1/country-itinerary/allocations
Request: { country: "italy", total_days: 10, group_type: "couple", vibes: [...], pacing: "moderate" }
Response: Two options with match_score, cities array with days per city
4
Day-by-Day Itinerary
120 seconds
Screen: Interactive Demo → Step 7 (Generate Itinerary)
Click "Generate Itinerary" (Step 7) — this is the big moment
This is the RAG pipeline in action. Three stages:
Stage 1 — Retriever: For each city, we generate a query embedding from the persona + vibes + destination, then run pgvector cosine similarity search to find the top POIs. We use 384-dimensional embeddings from BAAI/bge-small-en-v1.5.
Stage 2 — Scorer: The retrieved POIs are re-ranked using 21-dimension persona multipliers. A "romantic foodie couple" will see Trattoria Luzzi ranked higher than a sports bar. Same city, completely different results based on who's traveling.
Stage 3 — Assembler: The scored POIs are assembled into a day-by-day plan. The assembler respects pacing rules (how many activities per day), time-of-day preferences (morning sightseeing, afternoon museums, evening dining), transit times between venues, and meal break slots.
Point out in the timeline: Activity names with images, time slots, city headers, types (sightseeing, food, museum). Note the realistic scheduling — no overlapping times, proper meal breaks.
The same query with "adventure nightlife solo" would produce completely different venues, different time slots, different pacing. The persona isn't a filter — it reshapes the entire itinerary.
POST /api/v1/country-itinerary/generate
Pipeline: Embedding → pgvector search → Persona scoring → Day assembly
Output: Array of days, each with activities (title, type, start_time, image, description)

Wishlist & Explore
45 seconds
Screen: Interactive Demo → Step 4 (Wishlist)
Click "Load Wishlist" (Step 4) if not done already
The wishlist stores places the user has saved across all trips. Data is denormalized for instant display — no joins needed. The mobile app uses optimistic UI: the heart icon toggles instantly, then syncs with the backend in the background.
Point out: 4 landmarks with images — Colosseum (Rome), Eiffel Tower (Paris), Fushimi Inari (Kyoto), Sagrada Familia (Barcelona). These span multiple countries.
On the mobile app, the Explore tab surfaces destination inspiration — trending cities with hero images, a travel-by-vibe grid, and popular multi-city routes. All powered by the same city metadata in our database.
GET /api/v1/wishlist/ — returns array with poi_name, city, country, image_url
POST /api/v1/wishlist/ — add item
DELETE /api/v1/wishlist/{id} — remove item

Technical Closing
30 seconds
Show the Swagger docs tab briefly, then close with the summary
Tech summary: No LLM — pure retrieval + scoring + assembly. pgvector with 384-dim embeddings from BGE-small. 21-dimension persona scoring per POI. Country planner with constraint satisfaction. Day assembler with pacing rules. Anonymous device sessions. Auto-saving drafts. FastAPI + async PostgreSQL, deployed on Google Cloud Run.
What's next: Sprint 3 adds the timeline view and map integration on the mobile app, review ingestion from Reddit and TripAdvisor, knowledge graph sequencing for smarter ordering, and AI output validation tests.
